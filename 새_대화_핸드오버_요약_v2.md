## 2. 사용자 작업 스타일 이해

### 기획서 작성 방식
1. **하이브리드 구조 선호**: UI 섹션 중심 + 속성별 파일 분리
2. **중복 내용 최소화**: 같은 정보가 여러 곳에 반복되지 않도록 구조화
3. **실행 우선**: 질문보다 먼저 작업 → 결과 보고 → 피드백 반영
4. **디테일 중시**: 번호 체계, 용어 일관성, 논리적 정합성을 철저히 준수
5. **보고 선행**: 대규모 변경 시 먼저 분석·보고 → 확인 후 실행 패턴 선호

### 피드백 스타일
- 직접적이고 구체적 ("이거 왜 이래?", "삭제해", "A로 해")
- 문제점 지적 시 즉시 수정 기대
- 여러 수정 사항을 번호로 나열하여 한 번에 지시하는 경우 많음
- 잘못 이해한 부분은 바로 정정해줌 — 정정 후 "이해한 내용 보고" 요청

### 의사결정 방식
- 옵션 제시 → 선택 → 선택한 것만 적용
- 선택 후 추가 수정 요청 빈번 (점진적 개선)
- 참고자료보다 **새로 결정한 내용이 항상 우선**
- 실현 가능성·논리적 정합성을 중요하게 여김 — 비현실적이거나 상충되는 내용은 보고 요청

### 작업 흐름 패턴
1. 사용자가 여러 항목을 번호로 지시
2. 에이전트가 분석·보고 (실행 금지 상태가 대부분)
3. 사용자가 확인·수정·추가 지시
4. 수차례 반복 후 "실행" 명령
5. 실행 후 정합성 검토 → 문제 발견 시 수정

## 3. 기술적 작업 시 주의사항

### PowerShell 한글 인코딩 문제
- Windows PowerShell에서 한글 폴더/파일명 처리 시 인코딩 오류 발생
- **해결 방법**: `Write` 도구로 파일 생성 (폴더 자동 생성됨)
- `Rename-Item`, `Move-Item`, `git mv` 등은 한글 경로에서 실패 가능

### 폴더/파일 변경 작업
- 폴더명 변경: 새 폴더에 파일 생성 → 기존 파일 삭제
- 파일명 변경: 새 파일 생성 → 기존 파일 삭제
- **주의**: 사용자가 수동으로 파일명을 변경한 경우, 기존 경로로 Write하면 중복 파일 생성됨. 반드시 현재 파일명을 Glob으로 확인 후 작업할 것.

### 파일 중복 방지
- 이전 에이전트에서 사용자가 수동 파일명 변경 후 에이전트가 구 경로로 Write하여 중복 파일 생성된 사례 있음
- 작업 전 반드시 Glob으로 현재 파일 목록 확인

## 4. 번호 체계 — 컴팩트 번호 체계 (★ 가장 중요)

### 4-1. 개요
- **유일한 활성 번호 체계**. 이전 체계(`1:`, `A.`, `a.`, `•` 등)는 폐기(Deprecated).
- `.cursor/rules/10-wf-biz-docs-auto.mdc.mdc` Section 10에 상세 규칙 정의됨.
- 이전 체계 규칙도 같은 파일에 [Deprecated] 표기로 보존됨 (기존 파일 이해용).

### 4-2. 핵심 원칙
- **자릿수 = 계층 깊이**: 각 문자 하나가 하나의 레벨
- **풀 패스(Full Path) 통일**: 폴더명, 파일명, 파일 내부 헤딩, 본문 항목 등 **모든 레벨**이 루트부터의 전체 경로 번호 사용
- **마침표**: 번호 끝에 마침표(.) 하나만 사용. 레벨 간 구분자 없음
- **무한 확장**: 14단계, 15단계 이후로도 동일 체계 계속

### 4-3. 값 범위
| 순번 | 표기 |
|------|------|
| 0 | `0` (개요 전용) |
| 1~9 | 숫자 그대로 |
| 10 | `A` |
| 11 | `B` |
| 12~ | `C`, `D`, `E`... 알파벳 순 |

### 4-4. 구조 예시 (728. 리워드•스탬프)
```
728. 리워드•스탬프/          ← 3단계 (폴더)
├── 7281. 현황/              ← 4단계 (폴더)
│   ├── 72810. 페이지 개요.md ← 5단계 (파일, 0=개요)
│   │   내부:
│   │   **72810.** 페이지 개요       ← 5단계 (볼드)
│   │   **728101.** 페이지 역할      ← 6단계 (볼드)
│   │   **7281011.** 페이지 정의     ← 7단계 (볼드)
│   │   72810111. 첫 번째 내용 항목  ← 8단계 (미볼드)
│   │   728101111. 하위 설명         ← 9단계 (미볼드)
│   │   7281011111. 더 하위          ← 10단계 (미볼드)
│   │
│   ├── 72811. 첫 번째 섹션.md ← 5단계 (파일)
│   └── ...
├── 7282. 내역/
└── 7283. 설정/
```

### 4-5. 서식 규칙
- **5~7단계** (파일 내 구조적 헤딩): `**번호.** 제목` 형태로 **볼드** 적용
- **8단계 이상** (본문 항목): 볼드 미적용, 컴팩트 번호만 표기
- **들여쓰기 없음**: 모든 레벨에서 들여쓰기 없음 (자릿수가 깊이를 나타냄)
- 줄끝 공백 2칸, 줄 간격 규칙 등은 기존과 동일

### 4-6. 적용 범위
| 범위 | 상태 |
|------|------|
| `728.` 이후 (리워드•스탬프~) | 처음부터 컴팩트 체계로 작성 |
| `7-1.` ~ `7-2-7.` | 기존 파일은 구체계로 존재. 수정 발생 시 해당 파일을 컴팩트로 전환 |

### 4-7. 기존 체계와 차이점 요약
| 항목 | 이전 (Deprecated) | 컴팩트 (현행) |
|------|-------------------|---------------|
| 폴더명 | `7-2-8-1.` (대시 구분) | `7281.` (연결) |
| 파일명 | `1.0.` (로컬 번호) | `72810.` (풀 패스) |
| 파일 내 헤딩 | `**1.0.1.1.**` | `**7281011.**` |
| 본문 항목 | `1:`, `A.`, `a.`, `•` | `72810111.`, `728101111.` |
| 들여쓰기 | 8단계+ 전각 공백 | 없음 |

3. **내용 보존**: 기존 내용 임의 삭제/요약 금지. 구조 변경만 허용.
4. **참고자료 < 새 결정**: 폴더 내 참고자료보다 사용자의 새 결정이 항상 우선
5. **실행 전 보고**: 사용자가 "실행 금지"라고 하면 분석·보고만. "실행"이라고 하면 작업.
6. **정합성 검토**: 작업 완료 후 관련 파일 간 내용 일관성·상충 여부 확인

### 과거 에이전트에서 발생한 실수 패턴
1. 사용자가 수동으로 파일명 변경한 것을 모르고 구 경로에 Write → 중복 파일 생성
2. "기존 체계 유지"를 "변경 안 함"으로 해석 → 사용자 의도는 "점진적 전환"이었음
3. 일부 레벨만 컴팩트 적용하고 나머지는 구체계 유지 → 사용자 의도는 "전 레벨 컴팩트"
4. 위 실수들은 사용자 피드백으로 모두 정정 완료됨
